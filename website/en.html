Turn Feishu into Codex's working entry point.

Codex Feishu sends Feishu messages into resumable Codex sessions. Projects stay routable, replies stay traceable, and runtime state stays operable. It is not a disposable bot script. It is an engineering bridge built to stay up.

Why this exists

Feishu handles communication. Codex handles execution. The bridge keeps context and control intact.

Route by project

Use `/project <alias>` to make the active repository explicit instead of re-explaining context in every message.

Resume by session

New work starts with `codex exec`; continued work resumes with `codex exec resume`.

Operate with confidence

Idempotency, native replies, audit logs, metrics, and runtime guards keep the bridge trustworthy over time.

System shape

A thin bridge. The critical control points still have to be exact.

The value is not only that Feishu can talk to Codex. The value is that every message goes through routing, session state, and runtime policy before it touches a working repository.

  • Feishu receives the message and provides the chat boundary.
  • The bridge resolves project, session, idempotency, and permissions.
  • Codex executes inside a project root with the expected shell, sandbox, and pre-exec setup.
  • Audit, metrics, and runtime state stay visible after the reply is sent.
Deployment posture

Two deployment shapes. Each solves a different operational problem.

Local-first

long-connection

No public callback URL required. Best for individual development, internal networks, and fast iteration.

  • Fast onboarding
  • Minimal deployment overhead
  • Strong fit for personal daily workflow
Shared service

webhook

Best for shared services, card callbacks, reverse proxies, and standard production exposure.

  • Card callback support
  • Reverse proxy and HTTPS friendly
  • Suitable for formal service deployment
