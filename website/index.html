How it works

不是简单转发消息，而是把工程上下文放到正确的控制点上。

01

消息先进入项目绑定层

先取当前 chat 绑定的项目；没绑定就回退默认项目，而不是让每条消息都重新猜仓库。

02

会话优先续当前上下文

已有 active session 就直接续；没有的话，可通过 /session adopt latest 接上本机最近会话。

03

仓库并发在执行前收口

同一仓库根目录命中全局串行锁时，会先给出 queued 状态和占用原因，而不是静默等待。

04

Codex 在明确边界内运行

项目根目录、sandbox、shell、pre-exec 和超时策略都在桥接层内显式管理。

05

回复后仍保留控制面

audit、metrics、run state、session history 和卡片操作都留在本地控制面，不依赖黑盒平台。

Capabilities

这轮迭代之后，首页应该反映的已经不是“能聊”，而是“能管”。

Project Routing

/project <alias> 会真正改写当前 chat 的默认项目

群里切一次，整群后续都落到同一个项目；私聊也会记住各自的当前项目。

Session Adoption

/session adopt latest|list|<thread_id>

飞书侧不再只能续桥接器自己保存的 thread，也能接管本机 Codex CLI 的原生 session。

Auto Adopt

切项目后可自动接上最近会话

开启 service.project_switch_auto_adopt_latest = true 后，项目切换会优先保留当前 chat 会话，否则自动接最近匹配的本地 session。

Runtime Discipline

queue key + project.root 双层串行

同项目 thread 不会乱写，不同群同时操作同一仓库也会被自动收口。

Visible Queue

排队不是黑盒

/status、状态卡片和普通回复都会显示 queued 原因，区分“当前项目已有任务”和“仓库正在被其他 chat 操作”。

Extended Context

Wiki、知识库、多媒体和记忆继续并到同一链路

文档检索、wiki 读写、图片文件音频元数据、thread summary 与 project memory 都能进入执行提示。
