# 飞书交互路线图

## 目标

把 `飞鹊 (Feique)` 从”飞书里的 AI 编码入口”继续升级成”飞书里的工程工作台”。下一阶段重点不是再加几个命令，而是把文档、知识、附件、语音和审批链真正接进现有的会话模型。

## 当前边界

当前版本已经稳定支持：

- 文本消息驱动 Codex CLI / Claude Code（`/backend` 会话级切换）
- 项目路由与会话续接（双后端）
- 原生 reply、卡片回调、审计与指标
- 本机 `long-connection` 和团队共享 `webhook`

当前还没有完整做完的是：

- 飞书文档 / 知识库对象的读取、创建、更新与权限回写
- 图片 / 文件 / 音频 / 视频消息的结构化理解与生成式回复
- 面向飞书卡片的审批、任务看板和多步骤工作流

## 用户画像

### 1. 个人开发者

- 在私聊里提问、切项目、续会话
- 偶尔上传截图、日志或文件让后端 CLI 看

### 2. 小团队负责人

- 在群里让机器人处理 issue、review 和回归
- 希望文档、知识库、运行状态都能回到飞书

### 3. 平台 / 运维团队

- 需要审计、指标、权限边界和审批链
- 需要把机器人变成长期可运营服务，而不是个人实验脚本

## 核心场景

### 场景 A：文档 / 知识库管理

- 让 Codex 读取飞书文档或知识库条目作为上下文
- 让 Codex 输出结果后直接生成或更新飞书文档
- 在群里通过卡片明确展示“写入文档”结果，而不是把状态散落在多条消息里

### 场景 B：多媒体信息沟通

- 用户发图片、文件、音频、视频
- Bridge 抽取结构化元信息，必要时下载内容并交给相应模型链路
- Codex 回复时既返回文本结论，也回传可下载成果或预览

### 场景 C：任务状态与审批

- 在飞书卡片里展示“运行中 / 已完成 / 已取消 / 失败”
- 需要高风险操作时，通过卡片做显式审批
- 在单条消息下串起“开始、回顾、落地”的闭环

## 竞品对照

### OpenClaw / 通用聊天机器人

- 强项在 IM 接入、长连接、卡片分发
- 弱项在本地工程上下文和 Codex / Claude Code 会话续接

### 飞书原生机器人 + 自定义后端

- 强项在飞书能力完整
- 弱项在代码工作流、项目路由、Codex 会话管理需要自己重做

### Codex App / CLI

- 强项在代码执行、会话上下文、工程能力
- 弱项在飞书消息、卡片、知识库、审批链并非原生入口

`飞鹊 (Feique)` 的定位仍然是中间层：用飞书承接沟通，用 Codex 或 Claude Code 承接执行，用桥接层承接状态和规则。

## MVP 边界

### 阶段 1：附件感知

- 识别 `image` / `file` / `audio` / `media` 消息
- 抽取文件名、类型、大小、发送者、时间戳
- 把附件摘要加入后端 prompt

### 阶段 2：文档与知识库

- 读取飞书文档链接并抽取正文
- 支持把最终结果落到飞书文档模板
- 支持从群聊消息发起“生成周报 / 变更说明 / 复盘”

### 阶段 3：多媒体闭环

- 图片理解
- 音频转写
- 语音播报回复
- 生成文件后回传飞书

### 阶段 4：审批与任务面板

- 卡片审批
- 任务状态回看
- 失败重试与人工接管

## 技术方案

### 文档 / 知识库

- 新增 `Feishu Docs Connector`
- 在配置里独立声明权限范围和允许操作
- 默认读多写少，写入过程走卡片状态回写

### 多媒体消息

- Bridge 把飞书消息统一归一成 `text + attachments + metadata`
- 文本仍走当前后端 CLI（Codex 或 Claude Code）
- 图片 / 音频 / 视频分别接图像理解、转写、语音等链路
- 最终结果仍回到同一条飞书会话

### 权限与审计

- 每类新能力单独加 scope 与 feature flag
- 写操作默认关闭
- 文档写入、文件回传、知识库更新都写审计

## 风险

- 飞书权限模型比纯文本消息复杂得多，scope 管理必须先做细
- 附件和文档经常包含敏感信息，下载、缓存和日志都需要收紧
- 多模态链路容易拖慢响应时间，必须把同步和异步回传分开
- 文档更新是高风险写操作，不能默认静默执行

## 下一步

1. 先实现附件感知的数据模型，不急着一次做完整下载和解析。
2. 然后做飞书文档读取与模板化回写。
3. 最后再把图片、音频、视频和卡片审批拼进统一工作流。
